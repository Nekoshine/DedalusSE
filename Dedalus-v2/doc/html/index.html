Subject : Let's play in the dedalus!

Dedalus

Le jeu Dedalus permet de voir des personnages avec une IA (intelligence artificielle) tenter de survivre dans le labyrinthe. Il peut y avoir plusieurs joueurs simultanés et leur but est de trouver le chemin vers le prochain niveau.

Le dernier niveau est particulier puisque avant d'aller à la sortie, les joueurs doivent tuer l'ensemble des Minotaures présents.

Malheureusement, les joueurs fatiguent à chercher la sortie et risquent de mourir de faim après un certain temps. En revanche, ils sont aidés par un fil d'Ariane et par un GPS qui indique la direction de leur cible : le Minotaure le plus proche ou la sortie la plus proche.

Le jeu et les Minotaures sont contrôlés par le Maître du jeu qui a pour objectif de ne laisser aucun joueur survivre.

Objectif du projet

Le but de ce projet n'est pas de réaliser la meilleure IA (ni pour les joueurs ni pour les Minotaures). Votre but est d'améliorer le moteur de jeu en utilisant des outils de programmation parallèle et de remplacement de code. À la fin, le jeu devra être plus automatisé et s'exécuter sur différents processus ayant chacun leur rôle à jouer.

Structure du rapport (pénalité : -5)

Le rapport devra être constitué d'aux plus 10 pages rédigées.

En plus des points précédents, un rapport technique, comme tout document, doit s’articuler autour de trois parties :

  • l’introduction qui présente le contexte, la problématique et le contenu du document ;
  • le développement qui correspond au cœur du rapport. En particulier, vous prendrez soin d'expliquer les différents choix de mise en œuvre et préciserez le fonctionnement de vos programmes ;
  • la conclusion qui synthétise le contenu du document et présente des perspectives ouvertes par les travaux présentés.

En particulier, l'introduction doit présenter le contexte technique de votre rapport (et pas que vous êtes à l'ENSSAT et que vous êtes en projet de systèmes d'exploitation 2). Il faut ensuite définir la problématique de votre étude, le but recherché et enfin annoncer le plan du rapport qui doit être lié à la structure de votre discours.

Quant à la conclusion, elle doit comporter une synthèse "globale" (c'est un pléonasme volontaire) du contenu du rapport et porter une analyse générale sur ce qui a été fait et observé. Elle doit se terminer en présentant des perspectives ouvertes par votre travail (impacts, points à confirmer, améliorations possibles, ...). Votre avis sur ce que vous avez appris à titre personnel, votre satisfaction ou insatisfaction personnelle à faire ce projet ou à suivre ce cours, vos difficultés d'organisation personnelle ("le chat a mangé mon ordinateur") ou toute autre considération sur l'âge du capitaine et le sens du vent sont superflus dans un rapport technique.

Matériel et outils disponibles

Pour ce projet, une version initiale du jeu est fournie. Vous trouverez une documentation des fonctions utilisables dans l'onglet "Files" de ce document.

Le code fourni utilise quelques outils qu'il peut être utile de connaître et d'utiliser :

  • Cette documentation est produite à l'aide de [Doxygen.](http://www.doxygen.nl/) Il vous est demandé de documenter aussi vos fonctions la même manière, en plus des commentaires usuels du code.
  • Attention, si vous êtes sur une machine trop ancienne (comme les machines de TP ENSSAT), vous ne pouvez pas utiliser build.sh. Vous devez utiliser le Makefile fourni. À la place, vous utiliserez [Make.](https://www.gnu.org/software/make//). Pour ce faire, utiliser les commandes suivantes pour compiler : 
     $ make
 $ make doc
  • le script build.sh ne fait que des appels basiques au moteur de production (build automation) [CMake.](https://cmake.org/) en utilisant le fichier de configuration CMakeLists.txt. En particulier, un fichier .c ou .h ajouté dans le répertoire ./src/ ou un de ses sous-répertoires sera automatiquement intégré à la chaîne de compilation.
  • Si vous observez les sources du programme principal dedalus.c, vous constatez que le code présent dans la fonction main n'est utilisé que pour analyser les différentes options (via la méthode read_parameters), charger les modèles et lancer le jeu. L'analyse des options est assistée par l'utilisation de la bibliothèque [getopt.](http://en.wikipedia.org/wiki/Getopt) Il vous est recommandé de l'utiliser également.

Utilisation de la démonstration

Pour compiler les sources, placez-vous dans le répertoire racine du projet et lancez la commande suivante : 

 	  $ ./build.sh

Le script de compilation comporte plusieurs options consultables avec la commande : 

 	  $ ./build.sh -h

Si vous n'avez pas la bonne version de Cmake (machine de TP ENSSAT) vous pouvez faire : 

 	  $ ./make

En l'état, cette opération produit un unique fichier binaire disponible dans le sous-répertoire bin et nommé "Dedalus". L'usage de ce binaire est décrit par une aide accessible grâce à l'option "-h". 

 	  $ ./bin/Dedalus -h

Un exemple d'utilisation du jeu pourra être : 

 	  $ ./bin/Dedalus -m ./data/map_mini_f_mult -M 40 -d 100000

De plus, un exemple de réalisation (au format binaire) des différentes versions attendues est disponible dans le répertoire "./bin". Il vous donnera des indications sur les comportements souhaités. Notez qu'ils ne définissent pas un canon et que vous pouvez proposer vos propres variations (et les explications adéquates dans le rapport). Ces binaires ont été compilés pour les machines de TP de l'ENSSAT. Ils ne sont probablement pas compatibles sur d'autres machines. Vous pouvez les tester comme suit : 

 	  $ ./bin/Dedalus-v4 -m ./data/map_mini_f_mult -M 40 -d 100000

La présente documentation peut être recompilée avec la commande suivante : 

 	  $ ./build.sh --doc

Si vous n'avez pas la bonne version de Cmake (machine de TP ENSSAT) vous pouvez faire : 

 	  $ ./make doc

Travail à réaliser

Jusqu'à cinq versions du jeu sont attendues. Elles sont triées par ordre de difficulté et chaque version est la suite de la version précédente. La note entre parenthèses vous indique votre note maximale si vous ne réalisez "que" la version associée et les versions précédentes parfaitement.

Version 0 (note maximale : 0/20)

La version 0 correspond au code qui vous est fourni.

Version 1 (note maximale : 8/20)

En vous appuyant sur la V0, vous ferez en sorte que l'utilisateur lançant le jeu n'ait plus besoin de l'option "-p". Un nombre d'affichage adéquat s'ouvrira automatiquement pour chaque joueur. Vous devrez fermer (tuer) automatiquement chaque terminal à la fin du jeu.

Voici des pistes/indices qui peuvent vous être utiles. Ce ne sont pas des consignes et vous pouvez vous en affranchir.

Aides :

  • Il est fortement conseiller d'utiliser "Xterm" comme terminal contenant chaque affichage.
  • Xterm dispose d'une option pour dimensionner et positionner la fenêtre ("-geometry")
  • la méthode character_init est capable de trouver directement si l'affichage doit se faire dans le processus fils d'un autre processus (Xterm vs Bash).
  • La modification des fichiers config.c; config.h; game.c et dedalus.c devrait être suffisante.
  • Pour tuer un processus de pid ID, vous pouvez faire appel au programme kill vu en cours.

Une démonstration est disponible : 

 	  $ ./bin/Dedalus-v1 -m ./data/map_mini_f_mult -M 40 -d 100000

Version 2 (note maximale : 12/20)

Cette version reprend les bases de la V1. En revanche, ici, les propositions de mouvement des différentes IA vont être réalisées en parallèle dans différents processus afin de gagner du temps. Les mouvements des personnages restent synchrones.

Aides :

  • On peut utiliser le code de retour d'un processus (qui est compris entre 0 et 255) pour retourner de façon encodée le mouvement du personnage et l'information de triche calculée par la fonction character_propose_move.
  • La modification du fichier game.c devrait être suffisante.
  • Une solution utilisant "waitpid" est plus simple mais fait que les joueurs déclarés en premier sont prioritaires en cas de conflit.
  • Une solution plus complexe utilisant "wait" permet de donner la priorité à l'IA la plus rapide.

  srand(time(NULL) * getpid() * rand());

Une démonstration est disponible : 

 	  $ ./bin/Dedalus-v2 -m ./data/map_mini_f_mult -M 40 -d 100000

Version 3 (note maximale : 16/20)

En adaptant la V2, proposez une version asynchrone du jeu. Ainsi, la boucle principale de game_start attend qu'un personnage indique le mouvement qu'il souhaite faire et elle le traite avant d'attendre une nouvelle demande d'un personnage. Une IA plus rapide pourra donc jouer plus souvent qu'une IA lente.

Aides :

  • Au lieu de traiter tous les personnages en même temps, il suffit de faire tous les traitements d'un personnage directement.
  • La modification du fichier game.c devrait être suffisante.
  • La gestion des conflits devrait être beaucoup plus simple.
  • Attention, une IA peut retourner un déplacement pour un personnage alors que celui-ci est mort en combat.
  • Il peut être pratique de vous définir une structure contenant un "lien" vers le personnage considéré, le pid du processus calculant son mouvement et le résultat du calcul.
  • Ne pas oublier de relancer une demande de mouvement uniquement pour le personnage qui vient de jouer, mais après avoir fait le mouvement.
  • Attention aux fuites mémoires.

Une démonstration est disponible : 

 	  $ ./bin/Dedalus-v3 -m ./data/map_mini_f_mult -M 40 -d 100000

Version 4 (note maximale : 20/20)

En adaptant la V3, faites en sorte qu'un combat n'empêche pas les autres personnages de se déplacer. Le combat est donc calculé et affiché par un processus dédié. Vous pourrez proposer une version simplifiée de cela en faisant en sorte que seul le premier combat soit non bloquant. Si un deuxième combat simultané devait avoir lieu, alors vous bloquez tout en attendant que le premier combat se termine. C'est cette version qui est présentée dans la démonstration. Vous avez évidemment de droit de proposer une solution sans cette restriction.

Aides :

  • Cette version est complexe sans utiliser des outils que vous ne verrez qu'en SE3.
  • La modification du fichier game.c devrait être suffisante (étonnamment).
  • Penser à conserver une liste des personnages qui combattent et le pid du processus calculant le combat.
  • Attention à ne pas faire se déplacer un personnage combattant.
  • il faut penser à retourner le résultat du combat au processus principal. Seuls les points de vie des deux participants suffisent et, bonne nouvelle, 100 + 100 < 255. Vous pouvez donc utiliser le code de retour d'un processus.
  • Attention, si vous avez d'autres processus lancés pour d'autres tâches (comme demander un mouvement) vous devrez vérifier au retour de chaque "wait" de votre programme que vous faites le bon traitement.

Une démonstration est disponible : 

 	  $ ./bin/Dedalus-v4 -m ./data/map_mini_f_mult -M 40 -d 100000

Version 5 (note maximale : 25/20)

Ici, chaque joueur peut interagir avec son terminal d'affichage propre. Cette interaction permet à chaque joueur de démarrer son personnage en début de partie sans attendre les autres, de continuer après un combat... Les éventuels Minotaures ne démarrent pas leur déplacement avant qu'au moins un joueur ait débuté la partie. Une version "ultime" permettrait même de faire une commander manuelle d'un joueur en utilisant les touches du clavier.

Aides :

  • la méthode display_wait_user peut être un bon point de départ pour capturer une entrée utilisateur depuis un flux.
  • Pas d'autres aides. C'est la version [Munchkin !](https://en.wikipedia.org/w/index.php?title=Munchkin_(role-playing_games))

Cette version est la seule ne disposant pas d'une démonstration.
